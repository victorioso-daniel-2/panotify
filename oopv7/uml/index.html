Overview

This documentation provides a detailed explanation of the PaNotify application's class structure through UML (Unified Modeling Language) diagrams. The diagrams visualize the relationships between classes, their attributes, methods, and how they collaborate to form the complete system.

Diagram Legend

Association - "has a" relationship (e.g., Course has Exams)
Dependency - "uses" relationship (e.g., Service uses Model)

Class Relationships

This section explains the key relationships between classes in the PaNotify system.

Inheritance Relationships

  • User → Student: Student is a specialized type of User with student-specific behaviors
  • User → Instructor: Instructor is a specialized type of User with instructor-specific behaviors

Association Relationships

  • Course ↔ Exam: A Course contains multiple Exams; an Exam belongs to one Course
  • Exam ↔ Question: An Exam contains multiple Questions; a Question belongs to one Exam
  • Student ↔ ExamResult: A Student has multiple ExamResults; an ExamResult belongs to one Student
  • Question ↔ StudentAnswer: A Question has multiple StudentAnswers; a StudentAnswer refers to one Question
  • Student ↔ Course: A Student enrolls in multiple Courses; a Course has multiple Students
  • Instructor ↔ Course: An Instructor teaches multiple Courses; a Course has one Instructor

Dependency Relationships

  • Service Classes → Model Classes: Service classes depend on model classes to perform operations
  • UI Classes → Service Classes: UI classes depend on service classes to access functionality

Implementation Relationships

  • StudentTab ← CoursesTab, ExamsTab, etc.: These classes implement the StudentTab interface
  • InstructorTab ← CourseManagementTab, ExamCreationTab, etc.: These classes implement the InstructorTab interface
